# Agents & Components

This document outlines the agents and functional components of the Agentic AI EHR-to-FHIR extraction pipeline.

## Pipeline Architecture

The workflow transforms unstructured clinical notes into valid FHIR JSON/Python objects through a multi-step, agentic pipeline.

### 1. Clinical Analyst Agent
- **Role:** Primary extraction and mapping agent.
- **Responsibility:** Parses raw clinical notes (txt) to identify clinical entities (e.g., conditions, medications, observations).
- **Goal:** Directly map identified entities to `fhir.resources` Python objects.
- **Key Tool:** **FHIR Doc Tool** — Used to ensure schema accuracy by providing real-time, deterministic access to HL7 FHIR documentation.

### 2. Standardizer/Coder (Functional)
- **Role:** Terminology mapping component.
- **Responsibility:** Replaces the need for a secondary LLM agent by using high-performance algorithmic querying.
- **Goal:** Map clinical terms to standard codes (SNOMED-CT, LOINC, etc.) via the **NLM Terminology Service API**.
- **Integration:** Provides the necessary coding context to the FHIR resources generated by the Clinical Analyst.

### 3. FHIR Validator
- **Role:** Quality assurance and enforcement.
- **Responsibility:** Validates the final FHIR Python objects against official FHIR specifications.
- **Goal:** Utilize the built-in validation of `fhir.resources` or a standalone validator (e.g., HAPI).
- **Feedback Loop:** Implements **Back-and-Forth Validation**. If validation fails, error logs are returned to the Clinical Analyst for iterative correction.

## Support Tools

### FHIR Doc Tool (CLI/MCP Server)
- **Description:** A local tool or Model Context Protocol (MCP) server populated with official HL7 FHIR documentation.
- **Purpose:** Provides the agents with a "source of truth" for FHIR resource structures, minimizing hallucinations and ensuring compliance with the specification.

## Workflow: Back-and-Forth Validation
1. **Extraction:** Clinical Analyst generates an initial `fhir.resources` object.
2. **Standardization:** Functional component attaches SNOMED-CT codes.
3. **Validation:** FHIR Validator checks the object for schema compliance.
4. **Iterative Refinement:** If errors are found, the Clinical Analyst receives the error report and regenerates the object until it passes validation.
